# v3 설계 — 아이 직접입력(로비 패드) → 강사 확인 → 원장 대시보드

- **작성일**: 2026-06-18 (rev. 아이 직접입력 흐름 반영)
- **상태**: 설계 확정 — 구현 계획 단계로 이행
- **전략**: B (v2 데이터 모델 유지 + v1 가벼움 + 성능 수정)
- **코드 위치**: 기존 `AI-ON` 리포에서 진화 (git·v2 코드 재활용)
- **선행 자산**: v1(`starkids-swim`, Vercel 운영 중) · v2(`AI-ON`, `lib/v2/*`·스키마 010~013) · 엑셀 「AI-ON 영법 단계표 v2.xlsx」

---

## 1. 배경 & 핵심 흐름 변경

세 가지 자산이 있다.

| | 정체 | 특징 |
|---|---|---|
| **v1** (`starkids-swim`) | Vercel 운영 중 | 가볍고 사용성 좋음. 영법/세부단계 3단계 구조. |
| **v2** (`AI-ON`) | 구현 중 | 커리큘럼 DB·144단계·measurements·baseline. 무겁고 느림. |
| **엑셀** (영법 단계표 v2) | 진도 내용 | 대분류(영법)/중분류(킥·팔·콤비)/소분류(발차기 5m·15m·25m…) + 측정·첫완주·비고. |

**가장 큰 변경 — 입력 주체가 강사에서 아이로 바뀐다.** 3단계 흐름:

```
[아이] 로비 패드에서 직접 입력 (임시 기록)
        ↓
[강사] 확인·수정 (누락 아이 추가, 통과 판정·측정, 오류 수정 → 확정)
        ↓
[원장] 대시보드 (확정 기준 전체 현황)
```

업무 분담 효과: 아이가 자기 오늘 한 내용을 직접 기록·리뷰하며 학습 주도성을 갖고, 강사는 입력 노동 대신 **확인·판정**에 집중한다.

## 2. 설계 원칙

1. **데이터 모델은 v2 유지.** 엑셀 소분류(144단계·counter·repeatable·측정)를 담으려면 풍부한 모델이 필수이며 이미 있다.
2. **화면은 v1식 가벼움 + 아이가 쓸 수 있게 상세·쉽게.** 큰 버튼·시각적·현재 칸만.
3. **아이 입력 10초.** 아이가 이름 탭 → 출석 + 오늘 한 단계 "했어요" + 바퀴수 → 끝. 입력 후/무입력 **10초 후 자동으로 이름목록 복귀**(다음 아이 준비 + 남의 기록 프라이버시).
4. **아이 입력 = 임시, 강사 확인 = 확정.** 아이는 *활동을 보고*하고, *통과 판정·측정*은 강사가 한다.
5. **append-only 유지.** 모든 입력은 이벤트 적재, 상태는 집계로 도출. v2 원칙 계승.
6. **낙관적 UI + 성능 수정** — §6.

## 3. 진도 데이터 = 엑셀 시드 (커리큘럼 버전1)

엑셀 「영법 단계표 v2」를 커리큘럼 버전1로 시드. 대분류→`strokes`, 중분류→`skill_tracks`, 소분류→`skill_steps`(라더 순서=엑셀 행 순서).

**소분류 비고 → `step_kind` 매핑:**

| 엑셀 패턴 | step_kind | 강사 확인 시 컨트롤 | 저장 |
|---|---|---|---|
| 5m / 15m / 25m 진행 (대부분) | `ladder` | `[통과]` 탭 | `skill_progress`(observed) + 스냅샷 |
| 첫완주 `O` + 측정(시간 / 시간+스트로크) | `ladder` + `is_first_completion` + `measure_spec` | 통과 시 측정 인라인 | + `measurements`(time_sec/stroke_count) |
| 마스터·50m·IM ("위아래 1·2바퀴, 6회마다 측정") | `repeatable` | `[▲▼ 바퀴]` + 6회마다 시간 | `measurements`(laps/distance_m, 주기 time_sec) |
| 턴 ("클릭마다 누적, 완성 버튼, 몇 회 만에 완성") | `counter` | `[연습 +1]` + `[완성]` | 연습=`measurements`(attempt), 완성=`skill_progress` |

> 통과 판정·측정은 **강사 확인 단계의 컨트롤**이다. 아이 화면은 §4.1처럼 "했어요"만. 시드는 기존 `lib/v2/curriculum-seed.ts`를 엑셀 기준으로 갱신. 출시 전 완벽화 금지 — 엑셀=버전1 초안, 쓰며 보강.

## 4. 화면 (역할별)

### 4.1 아이 패드 (키오스크) `(/kiosk/[instructorId])`
- **패드 구성**: 강사별 패드, 갯수 설정 가능. 각 패드는 그 강사의 그 시간 아이만 표시.
- **이름목록**: 요일·시간대별 지금(또는 다음) 시간 아이들, **매시간 최대 5명**. 큰 이름 버튼.
- 아이가 자기 이름 탭 → **개인 입력 화면**:
  - 출석 (자동 출석, 큰 확인)
  - **오늘 한 단계 "했어요"** — 현재 소분류 1개를 크고 쉽게(그림/색). 형제 칩으로 "더 했어요" 선택 가능.
  - 바퀴수 (큰 ▲▼ 또는 숫자)
  - 저장 → "잘했어요!" + 오늘 한 내용 리뷰 한 줄.
- **자동 복귀**: 저장 후 또는 무입력 10초 → 이름목록으로. 키오스크 잠금(다른 화면 못 감, 남의 기록 안 보임).
- **인증**: 패드는 키오스크 계정(=해당 강사 스코프)으로 1회 로그인. 아이는 로그인 없이 이름 탭만. RLS로 그 강사 학생만 read/write.
- 저장 결과 = **임시 기록**(`source='child'`, 미확인).

### 4.2 강사 확인 `(/instructor/today)`
- 오늘 자기 반 아이 카드: 아이 임시입력 표시(출석·한 단계·바퀴수) + 상태 배지(미확인/확정).
- 액션: **[확인]**(확정) / 통과 판정·측정 입력(§3 컨트롤) / 오류 수정 / **미입력 아이 추가**.
- 현재 위치 자동 프리필(첫 미통과 ladder). 낙관적 UI.
- 확정 시 `sessions.status='confirmed'`, 필요 시 `skill_progress`/`measurements` 적재.

### 4.3 원장 대시보드 `(/director/dashboard)`
- **확정 기준** 전체 현황: 영법별 학생 분포 보드, 오늘 출결·입력 현황(미확인 잔여), 최근 통과·완주, 정체 학생.
- v1 director 화면 패턴 재사용.

## 5. 데이터 모델 (v2 위 최소 추가)

기존 v2 스키마(students/strokes/skill_tracks/skill_steps/skill_progress/measurements/sessions) 유지 + 확인 워크플로우용 컬럼 추가.

- **`sessions` 추가**: `input_source` (`child` / `instructor`), `status` (`pending` / `confirmed`), `confirmed_by` (FK profiles, null), `confirmed_at` (null), `reported_step_id` (FK skill_steps, null — 아이가 "했어요" 한 단계).
- **아이 입력 저장**: 당일 `sessions`(attendance='출석', input_source='child', status='pending', reported_step_id) + `measurements`(laps). **통과/측정 안 함**.
- **강사 확인**: `status='confirmed'`, confirmed_by/at 기록. 필요 시 `passStep`(observed)·측정 추가. 미입력 아이는 강사가 session 생성(input_source='instructor').
- **append-only**: 아이·강사 입력 모두 이벤트. 확정은 상태 전이(session에 한정). 진도/측정은 append.
- 무료 유지 — 기존 Supabase 프로젝트 내 신규.

## 6. 느림 해결 (전략 B의 나머지 절반)

| 조치 | 효과 |
|---|---|
| **Supabase 리전 서울(ap-northeast-2) 확인·이전** | 쿼리 왕복 지연 최대 감소 (의심 1순위) |
| **커리큘럼 캐싱** — 단계표 거의 불변 → 빌드타임 상수/`unstable_cache` | 매 입력마다 DB 왕복 제거 |
| **낙관적 UI** | 아이·강사 탭 즉시 반응, 콜드스타트 체감 제거 |
| **스트리밍 SSR(Suspense)** | 첫 페인트 즉시 |
| **PWA + 키오스크** | 패드 홈화면 설치·정적 캐시·전체화면 잠금 |

> 구현 전 배포본에서 느린 구간(초기 HTML SSR DB대기 vs 입력 응답) 실측해 우선순위 확정.

## 7. 범위 & 단계

- **Phase 1 (v3 MVP)**: ① 엑셀 커리큘럼 시드 갱신 ② 아이 키오스크 입력(이름목록·현재단계·바퀴수·10초 자동복귀) ③ 강사 확인(확정·통과·측정·수정·미입력 추가) ④ 원장 대시보드(확정 기준 현황) ⑤ 베이스라인 1회 배치 ⑥ **성능 수정**.
- **Phase 2~**: 정체 감지·부모 리포트·성장 예측·강사 성과·운영 분석.

**버리는 것**: v2 전체 사다리 화면(144개 일괄 렌더) · v1 3단계 데이터 모델(소분류 못 담음).

## 8. 플랫폼 결정

**Vercel PWA 유지.** 로비 패드 = PWA 키오스크(홈화면 설치·전체화면). 강사 폰·원장 어디서나 웹. 네이티브 앱은 심사·별도 코드베이스 부담만 늘고 지금 문제(느림=설정)를 안 풀어줌. 필요 시 Capacitor 래핑(후순위).
